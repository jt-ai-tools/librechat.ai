## fileStrategy (檔案策略)

- **選項**："local" | "firebase" | "s3" | "azure_blob"

<OptionTable
  options={[
    ['fileStrategy', '字串', '決定使用者上傳/產生的檔案儲存位置。若省略則預設為 `"local"`。', 'fileStrategy: "firebase"' ],
  ]}
/>

- **備註**：
  - `"firebase"` 是目前唯一透過真實 CDN (內容傳遞網路) 提供檔案的選項，這意味著圖片與頭像將從全球邊緣節點傳遞。S3 與 Azure Blob Storage 是物件儲存後端 —— 檔案可被存取，但預設不具備 CDN 加速。
  - S3 透過 **預簽名 URL (presigned URLs)**（臨時簽署權杖）提供檔案，該 URL 會過期。一旦過期，UI 中引用該 URL 的任何圖片或頭像都會顯示損壞。這使得 S3 不適合用作視覺資產的主要策略。詳情請參閱[相關討論](https://github.com/danny-avila/LibreChat/discussions/10280#discussioncomment-14803903)。
  - CloudFront (針對 S3 的 CDN) 已在計畫中並積極開發中。
  - 為了讓圖片與頭像獲得最佳效能，請使用 `"firebase"` 或配置 `fileStrategies` 將 `avatar` 與 `image` 路由至具備 CDN 支援的策略。
  - 設定詳情請參閱[檔案儲存與 CDN 文檔](/docs/configuration/cdn_zh_TW)。

## fileStrategies (細粒度檔案策略)

允許針對不同檔案類型進行細粒度的儲存策略控制。
- **可用策略**："local" | "firebase" | "s3" | "azure_blob"

<OptionTable
  options={[
    ['fileStrategies', '物件', '為不同的檔案類型配置不同的儲存策略。比單一 fileStrategy 選項更具彈性。', ''],
  ]}
/>

**子鍵 (Sub-keys)：**

<OptionTable
  options={[
    ['default', '字串', '當未定義特定類型時的備援儲存策略。預設為 "local"。', ''],
    ['avatar', '字串', '使用者與智慧體頭像圖片的儲存策略。建議使用具備 CDN 支援的策略 (`"firebase"`) 以獲得最佳效能。', ''],
    ['image', '字串', '聊天中上傳圖片的儲存策略。建議使用具備 CDN 支援的策略 (`"firebase"`) 以獲得最佳效能。', ''],
    ['document', '字串', '文件上傳（PDF、文字檔等）的儲存策略。', ''],
  ]}
/>

- **備註**：
  - 此設定的優先權高於單一的 `fileStrategy` 選項。
  - 若未配置特定檔案類型，將退而使用 `default`，接著是 `fileStrategy`，最後是 `"local"`。
  - 圖片與頭像需要持久且穩定的 URL 才能在 UI 中正確呈現。S3 預簽名 URL 會過期（AWS 限制：IAM 使用者為 7 天，臨時憑證僅數小時），這會導致模型選擇器與聊天 UI 中出現損壞的圖片。詳情請參閱[相關討論](https://github.com/danny-avila/LibreChat/discussions/10280#discussioncomment-14803903)。請為 `avatar` 與 `image` 使用 `"firebase"`（目前唯一的 CDN 支援策略）以避免此問題。
  - 針對 S3 的 CloudFront 支援正在開發中，將可消除預簽名 URL 的限制。
  - S3 與 Azure Blob Storage 非常適合 `document` (文件) 儲存，因為短效的預簽名下載 URL 對此類場景是合適的。
  - 各儲存供應商的設定詳情請參閱[檔案儲存與 CDN 文檔](/docs/configuration/cdn_zh_TW)。

**範例：**

```yaml filename="fileStrategies - 集中設定"
# 為所有檔案類型使用單一策略
fileStrategies:
  default: "s3"
```

```yaml filename="fileStrategies - 混合策略"
# 將圖片與頭像路由至 CDN，將文件保留在物件儲存中
fileStrategies:
  avatar: "firebase"   # 頭像透過 CDN 傳遞
  image: "firebase"    # 產生/上傳的圖片透過 CDN 傳遞
  document: "s3"       # 文件使用物件儲存
```

```yaml filename="fileStrategies - 部分配置"
# 僅配置特定類型，其他使用預設值
fileStrategies:
  default: "local"
  avatar: "firebase"   # 僅頭像使用 Firebase CDN，其他皆為本地儲存
```

## clientImageResize (用戶端圖片縮放)

**Key：**
<OptionTable
  options={[
    ['clientImageResize', '物件', '配置用戶端圖片縮放，以優化檔案上傳並防止因圖片過大導致的上傳錯誤。', ''],
  ]}
/>

**子鍵 (Subkeys)：**
<OptionTable
  options={[
    ['enabled', '布林值', '啟用或停用用戶端圖片縮放功能。預設值：false。', 'enabled: true'],
    ['maxWidth', '數字', '縮放後圖片的最大寬度（像素）。寬度超過此值的圖片將被縮放。預設值：1920。', 'maxWidth: 1024'],
    ['maxHeight', '數字', '縮放後圖片的最大高度（像素）。高度超過此值的圖片將被縮放。預設值：1080。', 'maxHeight: 768'],
    ['quality', '數字', 'JPEG 壓縮品質 (0.1 到 1.0)。較高的值代表品質較好但檔案較大。預設值：0.8。', 'quality: 0.9'],
    ['compressFormat', '字串', '壓縮圖片的輸出格式。選項："jpeg"、"webp"。預設值："jpeg"。', 'compressFormat: "webp"'],
  ]}
/>

**說明：**

`clientImageResize` 配置支援在上傳前自動進行用戶端圖片縮放。此功能有助於：

- **防止上傳失敗**：避免大圖檔案超過伺服器限制。
- **減少頻寬使用**：在傳輸前壓縮圖片。
- **提升上傳效能**：縮小檔案體積。
- **維持圖片品質**：在優化檔案大小的同時兼顧品質。

啟用後，超過指定 `maxWidth` 或 `maxHeight` 尺寸的圖片將在上傳前於用戶端自動縮放。縮放會維持原始寬高比，同時確保圖片符合指定範圍。

**範例：**
```yaml filename="clientImageResize"
fileConfig:
  clientImageResize:
    enabled: true
    maxWidth: 1920
    maxHeight: 1080
    quality: 0.8
    compressFormat: "jpeg"
```

**備註：**

- 僅適用於圖片檔案 (JPEG, PNG, WebP 等)。
- 當圖片超過指定尺寸時會自動觸發縮放。
- 縮放過程中會保留原始寬高比。
- 該功能適用於所有支援的圖片上傳終端。
- 品質 (Quality) 設定僅適用於 JPEG 與 WebP 格式。
- 將品質設定過低 (< 0.5) 可能導致肉眼可見的畫質下降。

參見：[檔案配置物件結構](/docs/configuration/librechat_yaml/object_structure/file_config_zh_TW)

## mcpSettings

**Key：**
<OptionTable
  options={[
    ['mcpSettings', '物件', '定義模型上下文協定 (MCP) 伺服器的全域設定。', ''],
  ]}
/>

**子鍵 (Subkeys)：**
<OptionTable
  options={[
    ['allowedDomains', '字串陣列', '指定 MCP 伺服器連接允許的網域清單。對於內部/本地位址為必填。', ''],
  ]}
/>

- **備註**：
  - 這是一項安全功能，旨在防止透過 MCP 伺服器濫用/誤用內部位址。
  - 根據預設，LibreChat 限制 MCP 伺服器連接到內部、本地或私有網路位址。
  - 使用本地 IP 位址或網域的 MCP 伺服器必須 **明確** 被允許。
  - 與所有 YAML 配置變更一樣，需要重啟 LibreChat 才能生效。
  - 支援網域、萬用字元子網域 (`*.example.com`)、Docker 網域以及 IP 位址。

**範例：**

```yaml filename="mcpSettings"
mcpSettings:
  allowedDomains:
    - "example.com"           # 特定網域
    - "*.example.com"         # 所有子網域
    - "mcp-server"            # 本地 Docker 網域
    - "172.24.1.165"          # 內部網路 IP
```

參見：[MCP 設定物件結構](/docs/configuration/librechat_yaml/object_structure/mcp_settings_zh_TW)

## mcpServers

**Key：**
<OptionTable
  options={[
    ['mcpServers', '物件', '定義模型上下文協定 (MCP) 伺服器的配置，允許在應用程式內動態整合 MCP 伺服器。', ''],
  ]}
/>

**子鍵 (Subkeys)：**
<OptionTable
  options={[
    ['<伺服器名稱>', '物件', '`mcpServers` 下的每個鍵代表一個個別的 MCP 伺服器配置，由唯一名稱識別。', ''],
  ]}
/>

- **備註**：
  - 初始化發生於啟動時，必須重啟應用程式才能使變更生效。
  - `<伺服器名稱>` 是每個 MCP 伺服器配置的唯一識別碼。
  - 每個 MCP 伺服器可使用四種連接類型之一進行配置：
    - `stdio`
    - `websocket`
    - `sse`
    - `streamable-http`
  - `type` 欄位指定與 MCP 伺服器的連接類型。
  - 若省略 `type`，將根據 `url` 或 `command` 的存在與格式設定預設值：
    - 若指定了 `url` 且以 `http` 或 `https` 開頭，`type` 預設為 `sse`。
    - 若指定了 `url` 且以 `ws` 或 `wss` 開頭，`type` 預設為 `websocket`。
    - 若指定了 `command`，`type` 預設為 `stdio`。
  - 其他配置選項包括：
    - `timeout`：MCP 伺服器請求的超時時間（毫秒）。決定等待工具請求回應的時間。
    - `initTimeout`：MCP 伺服器初始化的超時時間（毫秒）。決定等待伺服器初始化的時間。
    - `serverInstructions`：控制是否將伺服器指令納入智慧體上下文。可以是 `true`（使用伺服器提供的）、`false`（停用）或自定義字串（覆寫伺服器提供的）。
    - `customUserVars`：（選填）定義個別使用者可為 MCP 伺服器設定的自定義變數（例如：API 金鑰、URL）。這些由使用者透過 UI 提供的值，可隨後在伺服器的 `headers` 或 `env` 配置中使用 `{{VARIABLE_NAME}}` 語法引用。這支援了 MCP 工具的按使用者身份驗證或自定義設定。
  - 參見：[MCP 伺服器物件結構](/docs/configuration/librechat_yaml/object_structure/mcp_servers_zh_TW)

**範例：**

```yaml filename="mcpServers"
mcpServers:
  everything:
    # type: sse # type 可選擇性省略
    url: http://localhost:3001/sse
    timeout: 30000
    initTimeout: 10000
    serverInstructions: true  # 使用伺服器提供的指令
  puppeteer:
    type: stdio
    command: npx
    args:
      - -y
      - "@modelcontextprotocol/server-puppeteer"
    timeout: 30000
    initTimeout: 10000
    serverInstructions: "請勿存取任何本地檔案或本地/內部 IP 位址"
  filesystem:
    # type: stdio
    command: npx
    args:
      - -y
      - "@modelcontextprotocol/server-filesystem"
      - /home/user/LibreChat/
    iconPath: /home/user/LibreChat/client/public/assets/logo.svg
  mcp-obsidian:
    command: npx
    args:
      - -y
      - "mcp-obsidian"
      - /path/to/obsidian/vault
  streamable-http-example:
    type: streamable-http
    url: https://example.com/mcp
    headers:
      Authorization: "Bearer ${API_TOKEN}"
    timeout: 30000
  per-user-crendentials-example:
    type: sse
    url: "https//some.mcp/sse"
    headers:
      X-Custom-Auth-Token: "{{USER_API_KEY}}" # 使用者提供 API 金鑰的佔位符，定義於下方的 `customUserVars`。
    customUserVars:
      USER_API_KEY:
        title: "服務 API 金鑰"
        description: "您在此服務的個人 API 金鑰。您可以從 <a href='https://example.com/api-keys' target='_blank'>此處</a> 獲取。"
    serverInstructions: true
```

參見：[MCP 伺服器物件結構](/docs/configuration/librechat_yaml/object_structure/mcp_servers_zh_TW)
