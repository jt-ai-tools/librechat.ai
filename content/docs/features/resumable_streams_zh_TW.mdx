LibreChat 具備具備韌性的串流架構，確保您永不遺失 AI 產生的內容。無論是連線中斷、切換分頁，或是在另一台裝置上接續，您的回應始終能被保存並保持同步。

## 為什麼這很重要

傳統的聊天應用程式在連線中斷時會遺失所有正在串流的內容。透過可恢復串流，LibreChat 可以：

- **保存每一則回應** —— 網路不穩、瀏覽器重新整理或伺服器重啟都不會導致數據遺失。
- **保持多個分頁同步** —— 在兩個瀏覽器分頁開啟同一個對話，觀察它們即時同步更新。
- **實現無縫裝置切換** —— 在電腦上開始對話，隨後在手機上繼續。
- **讓您自由地多工處理** —— 開始產生內容後，可以瀏覽其他分頁，回來後即可看到完整的回應。

## 運作原理

當您向 AI 模型發送訊息時，LibreChat 會建立一個追蹤所有串流內容的產生工作 (generation job)。當連線中斷時，神奇的事情發生了：

1. **自動偵測** —— 用戶端會立即偵測到斷線。
2. **狀態重建** —— 重新連線後，伺服器會重建所有先前已串流的內容。
3. **無縫同步** —— 遺失的內容會透過同步事件傳遞。
4. **透明接續** —— 串流從當前位置繼續進行。

這一切都是自動發生的 —— 無需使用者進行任何操作。

### 多分頁與多裝置體驗

可恢復串流最強大的面向之一是 **即時同步**：

- **同一個對話，多個視窗** —— 在兩個瀏覽器分頁開啟一個對話，兩者都會同時收到更新。
- **跨裝置連續性** —— 在筆記型電腦上開始一個長篇產生任務，隨後在手機上檢查結果。
- **團隊協作** —— 在共享對話中，所有檢視者都能即時看到內容出現。

## 部署模式

LibreChat 支援兩種部署配置：

### 單實例模式 (預設)

使用 Node.js `EventEmitter` 進行發佈/訂閱 (pub/sub) 的記憶體儲存。非常適合：
- 本地開發
- 單一伺服器部署
- Docker Compose 設定

**無需配置** —— 開箱即用。

### Redis 模式 (生產環境)

使用 Redis Streams 與 Pub/Sub 進行跨實例通訊。對於下列情境至關重要：
- 水平擴充部署
- 具備負載平衡的生產環境
- 高可用性設定
- Kubernetes 叢集

在 Redis 模式下，使用者可以在一部伺服器實例上開始產生內容，並在另一部實例上無縫恢復 —— 非常適合滾動更新部署與自動擴充 (auto-scaling)。

**備註：** 若您僅執行單個 LibreChat 實例，為可恢復串流使用 Redis 通常是不必要的 —— 記憶體模式即可處理一切。當您在負載平衡器後方擁有多個 LibreChat 實例時，Redis 就變得非常有價值。話雖如此，Redis 對於其他功能（如快取與工作階段儲存）在單實例部署中仍然很有用。

## 配置

### 啟用 Redis Streams

當啟用 Redis 時 (`USE_REDIS=true`)，可恢復串流會自動使用 Redis。您也可以明確地啟用它：

```bash filename=".env"
USE_REDIS=true
REDIS_URI=redis://localhost:6379
# 當 USE_REDIS=true 時，可恢復串流將自動使用 Redis
# 若要明確控制：
USE_REDIS_STREAMS=true
```

### Redis 叢集支援

針對 Redis 叢集部署：

```bash filename=".env"
USE_REDIS_STREAMS=true
USE_REDIS_CLUSTER=true
REDIS_URI=redis://node1:7001,redis://node2:7002,redis://node3:7003
```

LibreChat 會自動使用 hash-tagged 鍵值，以確保多鍵操作維持在同一個叢集槽位 (cluster slot) 內。

## 使用情境

### 不穩定的網路
在 WiFi 訊號不佳或行動網路環境下，當連線恢復時，回應會自動恢復。無需重新發送提示詞。

### 行動裝置使用者
在 WiFi 與行動數據間切換（或反之）而不會遺失回應。串流會從中斷處精確接續。

### 耗時較長的產生任務
對於會產生長篇回應的複雜提示詞，您可以放心查看其他分頁或應用程式。回來時回應將已備妥。

### 跨裝置工作流
在工作電腦上開始對話，通勤回家後在手機上檢查結果 —— 完整的回應就在那裡。

### 生產環境部署
橫向擴展至多個伺服器實例，同時維持串流連貫性。滾動部署不會中斷進行中的產生任務。

## 技術細節

### 內容重建 (Content Reconstruction)

系統彙整所有串流的 delta 事件以重建：
- 訊息內容（文字、工具呼叫、引用）
- 智慧體執行步驟與中間推理過程
- 元數據與狀態資訊

### 效能優化

**記憶體優先方案**：重新連線至同一伺服器實例時，LibreChat 會使用本地快取實現零延遲內容恢復，避免不必要的 Redis 往返。

**自動清理**：在查詢期間會移除過時的工作條目以防止記憶體洩漏。已完成的串流會自動過期。

**高效儲存**：記憶體模式使用 `WeakRef` 進行圖狀儲存，當對話結束時可實現自動垃圾回收。

### 數據流 (Data Flow)

| 組件 | 儲存機制 |
|-----------|-------------------|
| 數據塊 (Chunks) | Redis Streams (`XADD`/`XRANGE`) |
| 工作元數據 | Redis Hash 結構 |
| 即時事件 | Redis Pub/Sub 頻道 |
| 過期處理 | 串流完成後自動設定 TTL |

## 測試可恢復串流

您可以驗證此功能是否運作正常：

1. 與任何 AI 模型開始一段串流對話。
2. **分頁測試**：在新的瀏覽器分頁開啟同一個聊天 —— 兩者應同步。
3. **中斷測試**：短暫關閉網路連線，隨後重新連線。
4. **導覽測試**：在串流中途離開該頁面，隨後返回。

在所有情況下，您都應該看到完整的回應，且無任何數據遺失。

## 疑難排解

### 串流未恢復？

**檢查 Redis 連通性：**
```bash
docker exec -it librechat-redis redis-cli ping
# 應回傳：PONG
```

**驗證環境變數：**
```bash
# 確保 USE_REDIS_STREAMS 已設定
echo $USE_REDIS_STREAMS
```

### 內容出現重複？

這通常代表用戶端版本不匹配。請確保您執行的是最新版本的 LibreChat。

### 單實例模式下記憶體用量過高？

已完成的串流會被自動回收。若您發現記憶體用量過高，請檢查：
- 是否有執行時間極長且未完成的串流。
- 是否有發生錯誤且未被正確清理的串流。
